# OpenMP Parallelization of OTI `mul` (and other elementwise ops)

## Status: Not yet implemented (as of 2026-03-31)

**Note**: The `get_all_derivs` threshold approach (`num_threads=omp_get_num_procs() if size > 2000 else 1`)
was validated on 2026-03-31. Results for WDEGP (n_train=20, DIM=20):
- Naive prange (no threshold): 4.10s / 10 calls — **1.9x slower** than single-threaded
- With threshold: 1.11s / 10 calls — **1.9x faster** than previous best (2.16s)
- The same threshold approach should work for `mul` via `#pragma omp parallel for if(size > 5000)`

## Background

After parallelizing `get_all_derivs` with Cython `prange`, the `oti.mul` function became
the dominant bottleneck in JetGP optimization (~35% of total `nll_and_grad` time at scale).

Profiling results (DEGP, n_train=200, DIM=20, kernel matrix 4200x4200, 3 calls to nll_and_grad):

| Function             | Time   | % of total |
|----------------------|--------|------------|
| `oti.mul`            | 8.32s  | 35%        |
| `cho_solve`          | 3.66s  | 15%        |
| `_assemble_kernel_numba` | 2.97s | 13%    |
| `get_all_derivs`     | 1.73s  | 7% (after prange optimization) |

## Architecture

The `mul` call chain is:

1. **Python/Cython dispatch** (`algebra.pxi`):
   `mul(lhs, rhs, out)` dispatches based on type to `__mul__OX__`, `__mul__rX__`, etc.

2. **Cython → C bridge** (`algebra_utils/mul.pxi`):
   Each typed variant calls a C function, e.g.:
   - `oarrm20n2_mul_OO_to(&arr1, &arr2, &res)` — OTI array * OTI array
   - `oarrm20n2_mul_rO_to(scalar, &arr2, &res)` — real scalar * OTI array
   - `oarrm20n2_mul_oO_to(&num, &arr2, &res)` — OTI scalar * OTI array
   - `oarrm20n2_mul_RO_to(&darr, &arr2, &res)` — real array * OTI array

3. **C elementwise loop** (`algebra_elementwise_to.c`):
   ```c
   void oarrm20n2_mul_OO_to(oarrm20n2_t* arr1, oarrm20n2_t* arr2, oarrm20n2_t* res){
       uint64_t i;
       oarrm20n2_dimCheck_OO_elementwise(arr1,arr2,res);
       for ( i = 0; i < arr1->size; i++){
           onumm20n2_mul_oo_to(&arr1->p_data[i], &arr2->p_data[i], &res->p_data[i]);
       }
   }
   ```

4. **Per-element OTI multiplication** (`onumm20n2_mul_oo_to`):
   This is a fully unrolled function generated by `cmod_writer.py` that computes the
   truncated product of two OTI numbers. For m=20, n=2 there are 231 coefficients per
   number, and the multiplication involves cross-terms between orders.

## Proposed Change

Add `#pragma omp parallel for` to the C elementwise loops:

```c
void {arr_func}_mul_OO_to({arr_type}* arr1, {arr_type}* arr2, {arr_type}* res){{
    uint64_t i;
    {arr_func}_dimCheck_OO_elementwise(arr1,arr2,res);
    #pragma omp parallel for schedule(static)
    for ( i = 0; i < arr1->size; i++){{
        {num_func}_mul_oo_to(&arr1->p_data[i], &arr2->p_data[i], &res->p_data[i]);
    }}
}}
```

Each iteration writes to independent memory (`res->p_data[i]`), so this is safe.

## Overhead Concern

OpenMP thread spawn overhead is ~10-50 microseconds. Whether this helps or hurts
depends on the array size:

| n_train | Matrix size | Elements | Work per mul call | OpenMP verdict |
|---------|-------------|----------|-------------------|----------------|
| 20      | 420x420     | ~1,200   | ~30 us            | SLOWER (overhead dominates) |
| 50      | 1050x1050   | ~7,500   | ~190 us           | Marginal |
| 100     | 2100x2100   | ~30,000  | ~750 us           | Faster |
| 200     | 4200x4200   | ~120,000 | ~3 ms             | Much faster |

### Mitigation options:
1. **Size threshold**: Only use `#pragma omp parallel for if(arr->size > 5000)`
2. **Environment variable**: `OMP_NUM_THREADS=1` to disable at runtime
3. **Compile-time flag**: Conditional `#ifdef USE_OMP_ELEMENTWISE`

Option 1 (size threshold) is recommended — it's automatic and handles both cases:
```c
#pragma omp parallel for schedule(static) if(arr1->size > 5000)
for ( i = 0; i < arr1->size; i++){
    ...
}
```
The `if()` clause in OpenMP runs single-threaded when the condition is false, with
near-zero overhead.

## Other Parallelization Candidates

The same pattern applies to ALL elementwise operations in `algebra_elementwise_to.c`:
- `neg_to`, `sum_*_to`, `sub_*_to`, `mul_*_to`, `div_*_to`, `gem_*_to`, `trunc_*_to`

And in `functions_to.c`:
- `exp_to`, `log_to`, `sin_to`, `cos_to`, `pow_to`, etc.

All follow the identical `for (i=0; i<size; i++) { scalar_op(&in[i], &out[i]); }` pattern.
